### 第1题 · 意图分类聚合 ⭐⭐

**题目**：统计5月AI处理的会话，按意图分类汇总会话量、平均满意度、转人工率，按会话量降序。

```sql
SELECT intent_category,
       COUNT(*) AS total_sessions,
       ROUND(AVG(user_satisfaction), 1) AS avg_satisfaction,
       ROUND(SUM(CASE WHEN transfer_to_human = 1 THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 1) AS transfer_rate
FROM conversations
WHERE ai_handled = 1
  AND dt BETWEEN '2026-05-01' AND '2026-05-31'
GROUP BY intent_category
ORDER BY total_sessions DESC;
```

**口述要点**：用CASE WHEN做条件聚合算转人工率。如果面试官追问"为什么转人工率高的类目满意度反而高"，说明转人工不一定代表体验差——用户可能遇到复杂问题，人工妥善解决后满意度反而更高。

### 第2题 · 多表关联找未解决 ⭐⭐

**题目**：找出转人工且最终"未解决"的会话，显示session_id、user_name、intent_category、handle_time_min。

```sql
SELECT c.session_id, u.user_name, c.intent_category, h.handle_time_min
FROM conversations c
JOIN human_tickets h ON c.session_id = h.session_id
JOIN users u ON c.user_id = u.user_id
WHERE c.transfer_to_human = 1
  AND h.resolution_status = '未解决';
```

**口述要点**：三表JOIN，通过session_id和user_id串联。面试时可补充：如果数据量大，可在WHERE条件里先过滤trans_to_human=1缩小范围。

### 第3题 · 分组TopN ⭐⭐⭐

**题目**：每个意图分类中响应时间最快的前3条会话。

```sql
SELECT intent_category, session_id, response_time_sec, ranking
FROM (
    SELECT intent_category, session_id, response_time_sec,
           ROW_NUMBER() OVER (PARTITION BY intent_category ORDER BY response_time_sec ASC) AS ranking
    FROM conversations
    WHERE ai_handled = 1
) t
WHERE ranking <= 3;
```

**口述要点**：ROW_NUMBER按意图分区、按响应时间升序排名。如果允许并列用RANK()或DENSE_RANK()。**必问**：ROW_NUMBER vs RANK vs DENSE_RANK 的区别——ROW_NUMBER不并列(1,2,3,4)，RANK跳过并列(1,1,3,4)，DENSE_RANK不跳过(1,1,2,3)。

### 第4题 · AB实验效果对比 ⭐⭐⭐⭐

**题目**：比较A组(旧版AI)和B组(新版AI)在5月的会话量、转人工率、平均满意度、AI解决率。

```sql
SELECT experiment_group,
       COUNT(*) AS session_cnt,
       ROUND(SUM(CASE WHEN transfer_to_human = 1 THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 1) AS transfer_rate,
       ROUND(AVG(user_satisfaction), 2) AS avg_satisfaction,
       ROUND(SUM(CASE WHEN resolved = 1 AND transfer_to_human = 0 THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 1) AS ai_resolution_rate
FROM conversations
WHERE dt BETWEEN '2026-05-01' AND '2026-05-31'
  AND ai_handled = 1
GROUP BY experiment_group;
```

**口述要点**：AI解决率 = AI独立解决(resolved=1且未转人工) / 总会话。这是AI客服的北极星指标。如果面试官问"B组转人工率低但满意度没提升怎么解读"——可能AI拦截了简单问题，但复杂问题的体验没有明显改善。

### 第5题 · 高于均值用户 ⭐⭐⭐

**题目**：找出会话次数高于全体用户平均水平的用户。

```sql
WITH user_sessions AS (
    SELECT user_id, COUNT(*) AS session_cnt
    FROM conversations
    GROUP BY user_id
)
SELECT u.user_name, u.region, us.session_cnt,
       ROUND((SELECT AVG(session_cnt * 1.0) FROM user_sessions), 1) AS overall_avg
FROM user_sessions us
JOIN users u ON us.user_id = u.user_id
WHERE us.session_cnt > (SELECT AVG(session_cnt * 1.0) FROM user_sessions)
ORDER BY us.session_cnt DESC;
```

**口述要点**：CTE先算每人会话数，再和全局均值比较。如果面试官问"有没有更简洁的写法"——可以用窗口函数AVG() OVER()替代子查询。

### 第6题 · CTE + 超均值分析 ⭐⭐⭐⭐

**题目**：找出处理时长高于本分类均值50%以上的异常会话。

```sql
WITH cat_avg AS (
    SELECT intent_category,
           AVG(response_time_sec * turn_count) AS avg_duration
    FROM conversations
    WHERE ai_handled = 1
    GROUP BY intent_category
)
SELECT c.session_id, c.intent_category,
       ROUND(c.response_time_sec * c.turn_count, 1) AS actual_duration,
       ROUND(ca.avg_duration, 1) AS cat_avg_duration,
       ROUND((c.response_time_sec * c.turn_count - ca.avg_duration) / ca.avg_duration * 100, 1) AS exceed_pct
FROM conversations c
JOIN cat_avg ca ON c.intent_category = ca.intent_category
WHERE c.response_time_sec * c.turn_count > ca.avg_duration * 1.5
ORDER BY exceed_pct DESC;
```

**口述要点**：CTE是面试考察重点，优势是可读性好、可复用。实际业务中这类分析用于发现AI在某些场景下的异常慢响应，帮助定位模型或知识库问题。

### 第7题 · LAG环比 ⭐⭐⭐⭐

**题目**：按天统计AI会话量，用LAG算日环比变化率，显示近14天。

```sql
WITH daily AS (
    SELECT dt, COUNT(*) AS ai_sessions
    FROM conversations
    WHERE ai_handled = 1
    GROUP BY dt
)
SELECT dt, ai_sessions,
       LAG(ai_sessions) OVER (ORDER BY dt) AS prev_day,
       ROUND((ai_sessions - LAG(ai_sessions) OVER (ORDER BY dt)) * 100.0
             / LAG(ai_sessions) OVER (ORDER BY dt), 1) AS change_pct
FROM daily
ORDER BY dt DESC
LIMIT 14;
```

**口述要点**：LAG(列, 偏移量, 默认值)，默认偏移1行。第一行LAG返回NULL需要处理。环比 = (当天-昨天)/昨天×100%。如果面试官问"还有哪些窗口函数"——LEAD(向后取)、FIRST_VALUE、LAST_VALUE、NTILE(分桶)。

---

### 第8题 · 滚动窗口+累计 ⭐⭐⭐⭐⭐

**题目**：基于daily_cost表，计算每日费用、累计费用、过去7天滚动均值。

```sql
SELECT dt, api_cost_cny,
       SUM(api_cost_cny) OVER (ORDER BY dt
           ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS cumulative_cost,
       ROUND(AVG(api_cost_cny) OVER (ORDER BY dt
           ROWS BETWEEN 6 PRECEDING AND CURRENT ROW), 2) AS rolling_7d_avg
FROM daily_cost
ORDER BY dt;
```

**口述要点**：UNBOUNDED PRECEDING表示从第一行到当前行→累计求和。ROWS BETWEEN 6 PRECEDING AND CURRENT ROW → 当前行+前6行=7天窗口。这是AI成本监控的核心SQL，直接对应JD里"量化AI ROI"的要求。

### 第9题 · 7日留存 ⭐⭐⭐⭐

**题目**：首次使用AI客服后7天内再次使用的用户比例。

```sql
WITH first_use AS (
    SELECT user_id, MIN(dt) AS first_dt
    FROM conversations
    GROUP BY user_id
),
return_users AS (
    SELECT DISTINCT f.user_id
    FROM first_use f
    JOIN conversations c ON f.user_id = c.user_id
    WHERE c.dt > f.first_dt
      AND c.dt <= DATE(f.first_dt, '+7 days')
)
SELECT COUNT(DISTINCT f.user_id) AS total_users,
       COUNT(DISTINCT r.user_id) AS retained_users,
       ROUND(COUNT(DISTINCT r.user_id) * 100.0 / COUNT(DISTINCT f.user_id), 1) AS retention_7d_pct
FROM first_use f
LEFT JOIN return_users r ON f.user_id = r.user_id;
```

**口述要点**：用LEFT JOIN而非INNER JOIN——保留所有首次使用的用户，确保分母正确。留存率是衡量AI产品用户粘性的核心指标。

### 第10题 · 用户分层 ⭐⭐⭐⭐⭐

**题目**：按会话次数分三层(新用户/活跃/重度)，统计每层用户数、占比、平均满意度、转人工率。

```sql
WITH user_stats AS (
    SELECT user_id,
           COUNT(*) AS session_cnt,
           AVG(user_satisfaction) AS avg_satis,
           SUM(CASE WHEN transfer_to_human = 1 THEN 1 ELSE 0 END) * 100.0 / COUNT(*) AS transfer_rate
    FROM conversations
    GROUP BY user_id
),
user_tier AS (
    SELECT *,
           CASE WHEN session_cnt = 1 THEN '新用户'
                WHEN session_cnt BETWEEN 2 AND 5 THEN '活跃用户'
                ELSE '重度用户'
           END AS tier
    FROM user_stats
)
SELECT tier,
       COUNT(*) AS user_cnt,
       ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER(), 1) AS pct,
       ROUND(AVG(avg_satis), 2) AS avg_satisfaction,
       ROUND(AVG(transfer_rate), 1) AS avg_transfer_rate
FROM user_tier
GROUP BY tier
ORDER BY MIN(session_cnt);
```

**口述要点**：两个CTE串联——先算用户级指标，再分层。占比用窗口函数SUM(COUNT(*)) OVER()算总数。分层分析是数据分析师的核心能力，能发现"重度用户满意度更高但转人工率也高"这类洞察。

## 三、AI数据分析专项（这个岗位的核心！）

### Q1: 如何搭建AI智能客服的核心指标体系？

> **回答框架**：

分三个层级：

**北极星指标**：AI解决率（AI独立解决/总会话）、AI对GMV的贡献。

**过程指标**：

- 效果：一次解决率、转人工率、用户满意度(1-5分)
- 效率：平均响应时间、平均会话轮数
- 覆盖：AI承接率、意图识别准确率

**诊断指标**：

- 按知识类别拆解解决率（定位AI弱项）
- 按用户分层看使用率（新老用户/国内外）
- 异常监控：重复提问率、负面反馈率

> 我在长城汽车就做过三层指标体系——结果指标、过程指标、诊断指标，思路完全一样。

### Q2: 如何量化AI的ROI？

> **回答**：

三个角度：

**成本节省** = AI替代人工的客服成本 - AI调用成本。计算：AI处理会话量 × 人工单次成本 - 日均API费用。

**效率提升** = 用户等待时间缩短 → 转化为留存/NPS提升。

**业务增量** = AI主动推荐带来的转化（比如对话中推荐房源→成交）。

> 在数势云创做LLM优化时，意图识别准确率从75%→90%，直接降低了人工转接率，这就是可量化的ROI。

### Q3: 对LLM应用的数据指标了解吗？

> **回答**：

- 效果：回答准确率、幻觉率、意图识别准确率
- 效率：首字延迟(TTFT)、端到端延迟、Token消耗
- 体验：点赞/点踩率、复问率（同一问题反复问=没答好）
- 成本：单次调用成本、日均Token消耗

> 数势云创做LLM意图识别时，核心看的就是准确率和任务完成率。这些思路直接迁移到智能客服。

### Q4: AI客服转人工率从15%升到25%，怎么排查？

> **回答（三步框架）**：

**第一步-数据验证**：确认口径没变、不是某类会话误归类、不是某个时段异常。

**第二步-维度拆解**：
- 按知识类别：哪个领域AI处理不了？
- 按用户群：新用户vs老用户？国内vs海外？
- 按时段：全天高还是某个时段高？
- 按版本：最近更新了模型/prompt？

**第三步-根因定位**：
- 模型问题→和算法团队review变更
- 知识库缺失→补充FAQ
- 意图识别错误→优化prompt/分类器
- 外部因素（新产品上线咨询暴增）→扩容

## 四、AB实验（JD明确要求）

### Q7: 如何设计AB实验验证新版AI模型？

> **回答（全流程）**：

**设计**：用户级分流（user_id哈希，50/50），实验组用新版AI，对照组用旧版。核心指标=AI解决率、转人工率。护栏指标=用户满意度（不能牺牲体验换效率）。

**样本量**：基于基线AI解决率 + MDE，用statsmodels算。假设解决率50%，想检出3%提升→每组约2000人。

**执行**：先AA验证分流均匀性，跑满2周覆盖完整周期。

**分析**：t检验核心指标，分层看新老用户/不同意图的效果差异。

---

### Q8: AB实验常见陷阱？（5个）

1. **辛普森悖论**：整体有效，分群无效 → 做分层分析
2. **SRM**：实验/对照比例偏离 → AA验证+卡方检验
3. **组间干扰**：两组用户互相影响 → 按城市/区域隔离
4. **多重比较**：看太多指标→假显著 → Bonferroni校正
5. **新奇效应**：短期虚高 → 延长周期，看累计曲线是否稳定

---

### Q9: P值<0.05就能上线吗？

> 不能只看P值。还要看：
> 1. 效果量够不够大（统计显著≠业务显著，0.1%的提升P<0.05但没业务价值）
> 2. 护栏指标有没有恶化
> 3. 分层效果是否一致（有没有对某类用户是负向的）
> 4. 实验周期是否覆盖完整业务周期

## 五、Python & 预测建模

### Q10: 你做过哪些预测模型？怎么做的？

> 长城汽车时做门店销量预测，对比了3个模型：
> - ARIMA：线性趋势好，但捕捉不了节假日
> - Prophet：自动处理节假日，可解释性高
> - XGBoost：加更多特征（促销、库存），精度最高
>
> 最终选了XGBoost + 时间特征 + 业务特征，网格搜索+5折交叉验证，准确率85%。

### Q11: 如果让你预测AI客服日均会话量？

> 时间序列方法：
> - 特征：历史滞后项、星期、节假日、留学申请季、推广活动
> - 模型：Prophet做baseline（季节性+节假日），XGBoost加业务特征
> - 评估：MAPE（平均绝对百分比误差）
> - 应用：客服人力调度和成本预估
