## 二、SQL题（必考 · 最高频）

### Q1：分组TopN — 每个城市商机数量前3的品牌

**核心要点**：窗口函数 `ROW_NUMBER()` + `PARTITION BY`

**话术：**

> 我会用窗口函数实现分组TopN：
>
> ```sql
> SELECT city, brand, opp_cnt
> FROM (
>     SELECT city, brand,
>            COUNT(*) AS opp_cnt,
>            ROW_NUMBER() OVER (PARTITION BY city ORDER BY COUNT(*) DESC) AS rn
>     FROM opportunity_table
>     GROUP BY city, brand
> ) t
> WHERE rn <= 3;
> ```
>
> 核心思路是先用 `GROUP BY` 聚合出每个城市每个品牌的商机数，再用 `ROW_NUMBER()` 在每个城市内按数量降序排名，最后取前3。如果允许并列，可以用 `RANK()` 或 `DENSE_RANK()` 替代。

### Q2：找未打标的商机（有A无B）

**核心要点**：`LEFT JOIN + IS NULL`

**话术：**

> ```sql
> SELECT o.opportunity_id, o.brand, o.store_name
> FROM opportunity o
> LEFT JOIN tag t ON o.opportunity_id = t.opportunity_id
> WHERE t.tag_id IS NULL;
> ```
>
> 这是经典的"有A无B"写法。如果数据量特别大，也可以用 `NOT EXISTS`，性能可能更好：
>
> ```sql
> SELECT o.opportunity_id, o.brand, o.store_name
> FROM opportunity o
> WHERE NOT EXISTS (
>     SELECT 1 FROM tag t WHERE t.opportunity_id = o.opportunity_id
> );
> ```

### Q3：重复数据去重

**核心要点**：`ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...)`

**话术：**

> ```sql
> WITH dedup AS (
>     SELECT *,
>            ROW_NUMBER() OVER (
>                PARTITION BY brand, city, store_name
>                ORDER BY update_time DESC
>            ) AS rn
>     FROM raw_data
> )
> SELECT * FROM dedup WHERE rn = 1;
> ```
>
> 按品牌+城市+门店分组，取更新时间最新的那条，去掉重复的旧数据。

### Q4：连续N天有记录的商机

**核心要点**：`LAG()` 或自连接 + `DATEDIFF()`

**话术：**

> ```sql
> WITH t AS (
>     SELECT DISTINCT brand, city, dt
>     FROM opportunity
> ),
> t2 AS (
>     SELECT *,
>            LAG(dt, 2) OVER (PARTITION BY brand, city ORDER BY dt) AS dt_2
>     FROM t
> )
> SELECT DISTINCT brand, city
> FROM t2
> WHERE DATEDIFF(dt, dt_2) = 2;
> ```
>
> 用 `LAG(dt, 2)` 取往前第2天的日期，如果和当前日期差2天，说明连续3天都有记录。

## 三、AB实验（必考 · 核心加分项）

### 📊 AB实验核心概念速查

#### 假设检验基础

| 概念 | 通俗理解 | 统计定义 | 面试话术 |
|:----:|---------|---------|---------|
| **原假设 H₀** | "新政策没用" — 默认认为没效果 | 实验组和对照组指标无显著差异（μ₁ = μ₂） | "原假设是'新政策无效'，我们的目标是用数据来推翻它" |
| **备择假设 H₁** | "新政策有用" — 我们想证明的结论 | 实验组和对照组指标有显著差异（μ₁ ≠ μ₂） | "备择假设是'新政策有效'，如果能拒绝原假设，就接受备择假设" |

> **记忆口诀**：原假设 = 保守/现状（没效果），备择假设 = 激进/变化（有效果）。AB实验的本质就是**收集证据推翻原假设**。

#### 核心指标速查

| 指标 | 通俗理解 | 统计意义 | 面试话术 |
|:----:|---------|---------|---------|
| **P值** | "结果是不是碰巧发生的？" | 在原假设为真的前提下，观察到当前结果（或更极端结果）的概率 | "P值<0.05，说明如果新政策真的无效，我们观察到这个结果的概率不到5%，所以有理由相信新政策有效" |
| **置信区间** | "效果大概在什么范围内？" | 在95%置信水平下，真实效果所在的区间范围 | "转化率提升了5%，95%置信区间是[2%, 8%]，说明真实效果大概率在2-8个百分点之间" |
| **统计功效 (Power)** | "如果真有效，我能检测出来的概率？" | 当真实效果存在时，正确拒绝原假设的概率 | "统计功效80%意味着如果新政策真的有效，我们有80%的概率能检测出来" |
| **MDE (最小可检测效应)** | "多小的效果我才能发现？" | 在给定样本量和显著性水平下，能检测到的最小效果大小 | "MDE是1%，意味着如果真实效果小于1%，我们的实验可能检测不出来，需要扩大样本量" |
| **第一类错误 (α)** | "假阳性" — 把无效当有效 | 原假设为真时错误拒绝的概率（通常取0.05） | "α=0.05，意味着每做20次实验，可能有1次是假阳性" |
| **第二类错误 (β)** | "假阴性" — 把有效当无效 | 原假设为假时错误接受的概率（通常取0.2） | "β=0.2，意味着如果新政策真的有效，有20%的概率我们没检测出来" |
| **SRM (样本比不匹配)** | "分流是不是出了问题？" | 实验组和对照组人数比例偏离预期分配比 | "实验前一定要做AA验证，用卡方检验检查分流是否均匀，避免SRM导致结论不可信" |
| **AA测试** | "实验前先验证分流是否均匀" | 在实验开始前，把两组都设为对照组，验证两组指标是否有显著差异 | "AA测试通过后，才能保证后续AB实验的结论是可靠的" |

#### 四格表速记（α、β、Power的关系）

| 真实情况 ↓ | 统计结论：不拒绝H₀（认为无效） | 统计结论：拒绝H₀（认为有效） |
|:----------:|:---:|:---:|
| **H₀为真（实际无效）** | ✅ 正确（1-α） | ❌ 第一类错误（α） |
| **H₁为真（实际有效）** | ❌ 第二类错误（β） | ✅ 统计功效（1-β） |

> **面试话术**："这个四格表是AB实验的底层逻辑。α控制假阳性风险（通常5%），β控制假阴性风险（通常20%），统计功效1-β就是80%。我们做实验就是在α和β之间做权衡——想降低假阳性，就要扩大样本量；想提高检测能力（功效），也要扩大样本量。"

### Q5：介绍你做的AB测试项目


**核心要点**：STAR法则 + 突出全流程参与

**话术：**

> 在长城汽车时，业务方想评估新的门店补贴政策是否有效，我参与了AB测试全流程：
>
> **设计阶段**：基于门店历史销量、门店规模做分层抽样，确保实验组和对照组在关键维度上均衡。用 `statsmodels` 计算了所需样本量，保证统计功效≥80%。
>
> **执行阶段**：实验前先做了AA测试，验证分流均匀性。实验周期跑了2周，避免新奇效应。
>
> **分析阶段**：用双样本t检验分析转化率差异，同时用Bootstrap做稳健性验证。还做了分层分析，看不同规模门店的效果是否一致，排除辛普森悖论。
>
> **结果**：新政策提升门店转化率约5%，p值<0.05，推动业务全国上线。

### Q6：AB实验有哪些常见陷阱？

**核心要点**：5个陷阱 + 一句话解决方案

**话术：**

> 常见陷阱有5个：
>
> 1. **辛普森悖论** — 整体看有效，分群看无效。→ **解决方案**：做分层分析，按用户群/门店类型分别看效果。
>
> 2. **SRM（样本比不匹配）** — 实验组和对照组人数比例偏离预期。→ **解决方案**：实验前做AA验证，用卡方检验检查分流是否均匀。
>
> 3. **组间干扰** — 实验组和对照组用户互相影响（比如同一个商圈的门店）。→ **解决方案**：按城市或区域做隔离，或者用聚类随机化。
>
> 4. **多重比较** — 看太多指标容易发现"假显著"。→ **解决方案**：预先确定核心指标和护栏指标，用Bonferroni校正。
>
> 5. **新奇效应** — 用户对新功能的新鲜感导致短期效果虚高。→ **解决方案**：延长实验周期，或者看累计效果曲线是否趋于稳定。

### Q7：如何确定AB实验需要多少样本量？

**核心要点**：4个参数 + 一句话公式

**话术：**

> 需要4个参数：
> 1. **基线转化率** — 比如当前10%
> 2. **最小可检测效应（MDE）** — 比如想检测出1%的提升
> 3. **显著性水平 α** — 通常0.05
> 4. **统计功效 1-β** — 通常0.8
>
> 公式：n ≈ (Z_α/2 + Z_β)² × [p₁(1-p₁) + p₂(1-p₂)] / (p₂ - p₁)²
>
> 实际工作中我一般直接用Python的 `statsmodels.stats.power.tt_ind_solve_power()` 一行代码搞定。

## 四、业务场景题

### Q8：商机采集量突然下降50%，怎么排查？

**核心要点**：三步框架（数据验证 → 维度拆解 → 根因定位）

**话术：**

> 我会按三步走：
>
> **第一步：数据验证**
> - 先确认是不是系统问题——采集接口有没有报错？数仓有没有延迟？
> - 再确认口径有没有变——商机定义改了？采集规则变了？
> - 最后确认是整体下降还是某个维度下降
>
> **第二步：维度拆解**
> - 按城市拆：是新开城少了，还是老城采集停了？
> - 按品牌拆：是某个大品牌没采集到，还是普遍下降？
> - 按渠道拆：线上采集少了，还是线下BD录入少了？
>
> **第三步：根因定位**
> - 系统问题 → 联系采集团队修复
> - 流程问题 → 优化采集SOP
> - 竞争因素（比如Keeta加入导致商家流失） → 调整采集策略

### Q9：如何评估商机质量？

**核心要点**：5个维度

**话术：**

> 我会从5个维度评估：
> 1. **转化率** — 商机最终转化为签约的比例，这是最核心的
> 2. **时效性** — 从采集到分发的平均时长，越快越好
> 3. **完整度** — 商机信息字段的完整率，字段越全质量越高
> 4. **准确率** — 打标分类的准确率，可以抽样人工验证
> 5. **ROI** — 该商机最终带来的GMV除以采集成本

### Q10：B端和C端数据分析有什么不同？

**核心要点**：决策链长度、数据量级、业务复杂度

**话术：**

> 我觉得主要有三点不同：
>
> 1. **决策链不同** — C端是个人决策，快且感性；B端是企业决策，链条长、参与角色多（老板、采购、运营），需要更长时间跟进。
>
> 2. **数据量级不同** — C端是海量用户行为数据，B端是相对少量但更复杂的商家数据，更注重数据质量和字段完整度。
>
> 3. **业务复杂度不同** — B端需要考虑商家分层、品类差异、区域差异，同一个策略在不同商家身上效果可能完全不同，需要更精细化的分析。

## 五、简历深挖

### Q11：销量预测具体怎么做的？

**核心要点**：3个模型对比 + XGBoost胜出

**话术：**

> 我对比了3个模型：
> - **ARIMA**：传统时间序列，适合线性趋势，但对节假日等外部因素捕捉弱
> - **Prophet**：能自动处理节假日效应，可解释性好，但精度一般
> - **XGBoost**：树模型，能加入更多特征，精度最高
>
> 最终选了XGBoost，特征包括：时间特征（月份、季度、节假日flag）+ 业务特征（库存量、促销力度、历史销量）。经过网格搜索调参和5折交叉验证，准确率达到85%。

### Q12：定价策略论文研究的是什么？

**核心要点**：3个创新点 + 结果

**话术：**

> 我的研究针对网约车平台的三方市场（平台、司机、乘客），核心创新点是：
> 1. **引入乘客有限理性** — 传统模型假设乘客完全理性，我考虑了乘客的信息选择性接受行为，更贴近现实
> 2. **构建差异化定价模型** — 不同乘客群体不同定价，而非统一定价
> 3. **设计PSO-MSA混合算法** — 解决非线性模型的求解难题
>
> 最终平台利润提升约10%，发表2篇SCI论文。

### Q13：在数势云创做的LLM优化具体做了什么？

**核心要点**：结构化提示词 + Few-shot

**话术：**

> 当时智能Agent产品的意图识别准确率只有75%，我做了两件事：
> 1. **结构化提示词** — 把原来一段式的prompt拆成"角色+任务+约束+输出格式"的结构化模板
> 2. **Few-shot示例** — 每个意图类别加了3-5个典型示例
>
> 优化后特定场景准确率从75%提升到90%。同时独立完成了追问模块的接口测试，确保功能稳定上线。
